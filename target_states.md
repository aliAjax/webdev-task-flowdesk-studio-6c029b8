# FlowDesk Studio 目标状态说明

## 1. 文档目的

本文档定义 `webdev-flowdesk-studio-001` 的可观察目标状态。它与
`task.md` 和 `rubric.json` 配合使用，说明评审人员在每个关键导航或
交互步骤后必须能够看到、操作和验证的结果。

依据材料：

- 数据采购要求：
  `https://hcnmzflgf8a7.feishu.cn/docx/RgMfdl52VojjBaxregFcU0B8nsg`
- 产品需求：`task.md`
- 任务元数据：`metadata.json`
- 验收标准：`rubric.json`
- 当前视觉基线：`screenshots/`

本文件遵循采购文档中的以下原则：

- 项目必须能在浏览器中稳定运行并被真实观察。
- 截图和视觉输入是验收证据，但不能替代行为断言。
- 不能只验收默认页面，必须覆盖加载、空、错误、成功、选中、禁用、
  编辑后和交互后状态。
- 核心结果必须通过 Playwright 断言、DOM snapshot、截图、console
  检查或确定性单元测试验证。
- 任一质量闸门不通过即阻塞交付，不能用其他状态的得分抵消。

## 2. 运行约定

| 项目 | 目标要求 |
| --- | --- |
| 应用类型 | React + TypeScript + Vite 纯浏览器应用 |
| 主要视口 | 1440 x 900 桌面端 |
| 数据来源 | 仅使用仓库内的 `mock-data/` |
| 后端与登录 | 不需要后端和真实登录 |
| 默认地址 | 以 `npm run dev` 实际输出的本地地址为准 |
| 状态作用域 | 同一浏览器会话内跨路由保持变更 |
| 刷新行为 | 完整刷新可以恢复初始 mock 数据，行为需在 `README.md` 中说明 |
| 控制台 | 不得出现未捕获异常、`console.error`、资源加载失败或 React 警告 |
| 布局 | 不得出现页面横向溢出、文字裁切、面板重叠或浮层遮挡操作 |

以下全部状态都必须在不使用个人账号、远程 API、付费服务或不可控实时
数据的情况下完成验收。

## 5. 页面与交互目标状态

### TS-01 - Dashboard 加载完成

**路由：** `/`

**前置条件：** 初始 mock 数据已载入。

**预期状态：**

- 标题 `流程运营总览` 可见。
- KPI 值由当前全局 store 计算，而不是在页面重复硬编码：流程总数 `12`、
  已发布 `7`、草稿 `3`、异常或超时实例 `22`，并展示平均审批耗时。
- 最近流程按 `updatedAt` 排序，并提供稳定的
  `data-testid="recent-workflow"` 定位方式。
- 发布检查区域列出需要处理的草稿流程。
- 异常实例卡片与 Runtime Monitor 使用相同的实例 ID 和状态。
- 点击最近流程后进入对应编辑器，流程 ID、名称、业务域、版本、节点和连线
  上下文保持一致。

**所需证据：**

- 使用 `kpi-*` test ID 的 KPI 数值断言。
- Dashboard 三个信息区域的 DOM snapshot。
- 与 `01-dashboard.png` 参考用途一致的截图。
- 从最近流程跳转到正确编辑器 URL 的断言。

### TS-02 - 流程库加载完成

**路由：** `/workflows`

**前置条件：** 初始 mock 数据已载入。

**预期状态：**

- 展示 12 条流程，字段包含名称、业务域、状态、版本、编辑人、更新时间、
  异常实例数、节点数和行操作。
- 搜索、状态筛选、业务域筛选和排序控件均可用，并提供可访问名称。
- 超长流程名称在名称单元格内有意截断，同时通过无障碍文本或 tooltip 提供
  完整名称。
- 新建、复制、归档和进入编辑器会真实更新全局 store，且不会制造版本历史
  或流程上下文冲突。
- 每行可通过 `data-testid="workflow-{id}"` 稳定定位。

**所需证据：**

- 所有表格字段和 12 条结果的断言。
- 搜索、筛选和排序后的结果集合断言。
- 新建、复制、归档和进入编辑器的操作轨迹。
- 与 `02-workflows.png` 对应的全页截图。

### TS-05 - 审批节点选中并编辑

**路由：** `/workflows/wf-1/edit`

**触发方式：** 选中 `data-testid="canvas-node-approval"`，把审批人来源从

**所需证据：**

### TS-07 - 校验错误双层反馈

**路由：** `/workflows/wf-1/edit`

**触发方式：** 在修复初始 Condition 规则前运行校验。

**预期状态：**

- Condition 节点变为 `校验失败` 并展示可见错误标记。
- `data-testid="problem-panel"` 包含 `条件分支规则未配置`。
- `data-testid="error-count"` 显示准确错误数量。
- 点击问题条目会选中并显示对应画布节点。
- 校验还必须拒绝：缺少 End、Start 不可达的必经节点、无法到达 End 的路径、
  孤立节点、空审批人、无效表单字段，以及少于两个有效出口的 Condition。
- 存在校验错误时，发布必须被禁用或阻止。

**所需证据：**

- 节点错误样式与问题条目一一对应的断言。
- 错误数量断言。
- 从问题条目定位节点的断言。
- 覆盖所有图结构和配置错误的校验单元测试。
- 错误状态截图。

- 不得有步骤显示 `当前停留`，流程图中不得存在 `.runtime-current`。
- 已完成实例可以使用完成样式标记 End，但不能把它表现为进行中的当前节点。

**所需证据：**

- 已完成状态和低风险一致性断言。
- 当前步骤和当前高亮数量均为零的断言。
- 已完成实例详情截图。

### TS-16 - 版本比较状态

**路由：** `/workflows/wf-2/versions`

**触发方式：** 以 V1 为基准、V2 为目标进行比较。

**预期状态：**

- 版本选择器列出当前流程所有真实版本。
- 比较结果展示新增 Notify 节点、删除节点、具体新增和删除连线、实际配置字段
  变化，以及两个版本的发布说明。
- 配置差异必须根据所选版本快照计算，不能使用固定展示文本。
- 交换 V1 和 V2 后，新增和删除方向随之反转。
- 即使连线总数不变，只要连接关系改变，也必须识别为重连变化。

**所需证据：**

- 节点、配置和具体连线差异的确定性单元测试。
- 页面 DOM 值与版本 fixture 的对照。
- 交换版本方向后的断言。
- 与 `06-version-history.png` 对应的截图。

### TS-17 - 历史版本恢复状态

**路由：** 从 `/workflows/wf-2/versions` 到 `/workflows/wf-2/edit`。

**触发方式：** 点击 `data-testid="restore-v1"`。

**预期状态：**

- 应用返回对应编辑器并公布 `已恢复版本 V1 为草稿`。
- 流程状态变为草稿。
- 画布完整恢复 V1 的节点、连线、位置和配置，V2 的 Notify 节点不存在。
- 恢复历史版本不会删除不可变的已发布历史。
- 再次进入 Version History 后，当前版本标记和草稿基线语义保持一致。

**所需证据：**

- 路由和成功消息断言。
- 恢复图结构与 V1 快照的深度相等断言。
- `canvas-node-notify` 数量为零的断言。
- 恢复后的版本历史完整性断言。

### TS-18 - 加载、失败、空和成功状态

**路由：** Dashboard、Workflows、Editor、Preview、Monitor 和 Versions 中
适用的页面。

**触发方式：** 使用确定性的本地 fixture 或测试专用 store seed，不得调用
远程服务。

**预期状态：**

- **加载：** 使用稳定 skeleton 或进度状态，内容出现时不得导致页面明显跳动。
- **空状态：** 空流程、无搜索结果、无表单字段、无实例和无可比较版本时，
  提供明确且可恢复的引导。
- **错误：** 本地加载、解析或校验失败时指出失败区域，并提供重试、重置或
  导航操作。
- **成功：** 保存、发布、恢复、复制和归档成功消息可被公布并正常消失，
  不得遮挡操作。
- **选中、焦点和禁用：** 键盘焦点可见；当前节点、路由、筛选器和版本清晰；
  禁用控件能够表达不可操作原因。

**所需证据：**

- 至少四类非默认状态截图。
- 加载、空、错误和成功状态的 DOM 语义断言。
- 键盘焦点和禁用控件检查。
- 1440 x 900 下的布局抖动和重叠检查。

## 6. 完整验收链路

以下步骤必须在一个 Playwright 测试中使用同一个浏览器 `page` 连续完成，
以真实验证跨路由状态联动：

1. 打开 Dashboard，从最近流程进入 `wf-1` 编辑器。
2. 选中 Approval，把审批人来源改为 `固定角色`，保存节点并保存草稿。
3. 运行校验，断言画布节点和问题面板同步展示错误。
4. 选中 Condition，配置金额规则，保存并重新校验。
5. 断言错误数量减少，且 Approval 名称和配置没有被修改。
6. 打开 Preview，输入 `8000`，断言进入高金额分支。
7. 返回 Editor，仅在校验通过后发布。
8. 在 Workflows 和 Dashboard 中断言状态、版本、更新时间、发布时间和 KPI
   的精确变化。
9. 进入 Runtime Monitor，打开异常实例，核对时间线和当前节点身份。
10. 进入 Version History，比较真实版本变化，恢复 V1，并断言恢复后的编辑器
    图结构。
11. 完整链路结束后检查 console error 和 page error。

把这些步骤拆成会重置 store 的独立测试，不能证明所要求的端到端状态转换。

## 7. 证据输出

最终验收产物应写入 `artifacts/` 或 `submission/`，至少包括：

- Playwright list 或 HTML 报告。
- 完整验收链路的 trace。
- TS-01 至 TS-18 中指定的状态截图。
- console 和 page error 检查结果。
- 校验逻辑、版本 diff 和 mock 数据一致性的单元测试输出。
- `rubric.json` 每个叶节点与证据文件之间的简短映射。

现有 `screenshots/` 文件可以复制到最终证据目录，但代码修改前生成的截图
不得作为修改后状态的完成证据。
